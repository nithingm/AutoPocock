# Topologically Sorted DAG Planning and Graph-Driven Ralph Loop Orchestration

## Problem Statement

The current operating system can generate a PRD, create an Issue DAG artifact, and run bounded execution work, but it still does not behave like a near-autonomous idea-to-execution system for a technical Solo Builder. Planning remains too weak at the exact point where autonomy needs the most structure: the DAG is not yet the clearly vetted execution authority, wave selection is not yet driven by topological and conflict-safe graph semantics, and Ralph loops are not yet compiled as explicit, inspectable execution artifacts from approved DAG nodes.

That leaves several operational gaps:

- a PRD can exist without a strongly validated execution graph
- a DAG can exist without being the real source of execution readiness
- parallelism is underspecified even though subagent concurrency is one of the main product goals
- loop execution shape is too implicit, making feasibility, approval, and recovery harder to trust
- integration failures and bug loops do not yet re-enter the graph automatically with enough safeguards

For the product to feel close to plug-and-play for a technical Solo Builder, the system needs a stronger planning compiler: a user-editable, topologically sound DAG that becomes the execution contract for graph-driven Ralph loops, while still preserving artifact transparency and explicit approval boundaries.

## Solution

Build the next planning phase around an execution-grade DAG that is generated from the PRD, editable by the user, validated for execution readiness, and used to drive graph-based Ralph loop orchestration.

The target operator flow should be:

- optional ideation through `/grill-me`
- code-aware clarification through `/grill-with-docs` when repo-specific planning is needed
- PRD creation and user approval
- DAG generation from the approved PRD
- DAG validation for structural tightness, dependency integrity, bounded scope, and parallel safety
- GitHub issue synthesis from actionable DAG nodes
- wave planning from topological order plus write-surface and conflict rules
- explicit Ralph Loop Spec generation for approved wave nodes
- preflight feasibility validation of the wave bundle
- execution of safe parallel work with branch-local pause behavior and full-run freezes only for shared-foundation failures
- deterministic graph mutation from completion, validation failure, bug-loop creation, review, QA, and escalation outcomes

The product should preserve these principles:

- the PRD stays the upstream planning artifact
- the DAG becomes the execution authority after it is vetted
- every field is system-generated by default and user-overridable
- the user does not need to manually fill out node metadata to use the system
- every actionable DAG node becomes a GitHub issue
- Ralph loops launch only from structurally valid, explicitly approved DAG nodes or approved waves
- loop execution remains bounded by declared node scope and write surface
- integration failures automatically create new graph-linked repair issues when safe

## Implementation Decisions

- Treat the PRD as the upstream planning artifact and the vetted DAG as the downstream execution authority.
- Keep `/grill-me` positioned as general ideation and `/grill-with-docs` positioned as code-specific planning before PRD or DAG generation.
- Require a PRD-tightness validation pass before DAG compilation. User approval is necessary but not sufficient when structural failure signals are present.
- Generate the DAG from the PRD and recompile it when the PRD changes. DAG recompilation should preserve compatible user edits, mark stale derived values, and emit a reconciliation diff between PRD intent and DAG structure.
- Treat the DAG as a real topologically sortable execution graph rather than a flat checklist or a decorative planning artifact.
- Keep the actionable DAG GitHub-native: every actionable node should compile into a live GitHub issue, with deterministic mappings between node identity and tracker identity.
- Use typed actionable nodes such as implementation, investigation, HITL, review follow-up, and bug-loop to avoid overloading all work into one generic issue shape.
- Keep the author-facing node contract lean. The system should generate most fields automatically and allow full user override rather than demanding heavy manual metadata entry.
- Track field provenance so generated values, user edits, approval-frozen values, and stale values can be distinguished cleanly.
- Compute execution waves from the topological frontier, then filter and prioritize that frontier using write-surface overlap, conflict rules, tracer-bullet priority, dependency unlock value, and estimated review cost.
- Treat maximum concurrent subagent count as runtime policy, not as a DAG semantic. The scheduler should respect a configurable concurrency ceiling rather than assuming a fixed platform limit.
- Represent write surfaces as stable logical ownership areas such as module or subsystem surfaces, with optional expansion into concrete file sets during execution planning.
- Enforce stricter parallel-safety rules for shared-foundation surfaces such as schema, shared contracts, build configuration, or other high-coupling areas.
- Compile a Ralph Loop Spec artifact for each approved DAG node before execution. That artifact should include scope, owned surfaces, acceptance criteria, verification plan, stop conditions, escalation rules, retry budget, and any allowed internal parallelism.
- Support two graph levels explicitly:
  - an outer project DAG that decides what can run next across slices and stages
  - an inner node-level Ralph execution plan that structures how one approved node is executed and verified
- Require wave-bundle approval by default. A wave bundle should include selected nodes, dependency and conflict reasoning, compiled loop specs, feasibility checks, and stop conditions.
- Allow risk-based escalation to per-loop approval only when a node exceeds predefined sensitivity thresholds.
- Run preflight feasibility validation against approved nodes and loop specs before launch. If one node is infeasible but others remain safe, split the wave dynamically and emit a reconciliation artifact rather than failing the whole bundle.
- Require completion evidence to map acceptance criteria to observed outputs, verification results, tests, and lightweight review evidence where applicable.
- Treat testing as multidimensional. The expected evidence shape should include unit, integration, end-to-end, or workflow-level tests depending on the node’s behavior and merge surface.
- Introduce an intermediate validation-failed state for work that executed but did not pass broader integration or review gates.
- When integration failures cannot be cleanly assigned to one original node, create new narrow bug-loop issues automatically, link them to the failed wave and source nodes, and insert them into the DAG with computed dependencies.
- Bound auto-repair behavior with explicit limits:
  - max 2 automatic same-node repair loops
  - max 2 automatic integration bug-loop generations per failed wave
  - max 3 total descendant repair issues from one original node before human review
  - force escalation on repeated failure of the same acceptance criterion
  - force escalation when a repair issue expands write surface beyond the original affected area
- Pause only the affected branch and its dependents when a node or repair path escalates. Freeze the whole run only when a shared foundation or wave-level assumption is invalidated.
- Define shared-foundation full-run freezes around events such as unstable shared schemas, unstable shared contracts, globally broken build or runtime environments, architecture contradictions, missed broad write-surface overlap, or incorrect dependency structure in the accepted DAG.
- Treat multi-stage “single run” execution as a chain of approved wave bundles and compiled node loop specs rather than one opaque mega-loop.
- Preserve artifact-first visibility in the console and repo by surfacing PRD-to-DAG diffs, graph waves, node provenance, loop-spec previews, feasibility findings, repair insertions, and branch-local versus full-run pause decisions.

## Testing Decisions

- A good test should verify externally visible planning and execution behavior rather than internal helper structure.
- Test the PRD-tightness validator against strong, weak, contradictory, and underspecified plans to confirm that the system blocks unsafe decomposition and emits actionable failure reasons.
- Test DAG recompilation against edited PRDs and user-modified DAGs to verify preservation of compatible edits, stale marking, and PRD-to-DAG diff generation.
- Test topological sorting and wave generation across graphs with different dependency shapes, conflict patterns, and write-surface overlaps.
- Test scheduler prioritization rules for tracer bullets, dependency unlock value, conflict isolation, and review-cost-aware tie-breaking.
- Test write-surface normalization and parallel-safety rules, especially around shared schema, shared contract, and build-configuration surfaces.
- Test loop-spec compilation from approved DAG nodes to verify that scope, acceptance criteria, retry budgets, escalation rules, and allowed inner parallelism remain faithful to node contracts.
- Test wave-bundle preflight validation, including dynamic wave splitting when one approved node becomes infeasible.
- Test completion progression with multiple evidence shapes, including unit, integration, end-to-end, and workflow-level verification expectations.
- Test validation-failed progression so locally complete but integration-broken nodes do not unlock dependents incorrectly.
- Test automatic bug-loop insertion from integration failures, including source-node linkage, dependency insertion, repair-scope narrowing, and escalation thresholds.
- Test branch-local pause behavior versus full-run freezes using fixture-backed shared-foundation failure scenarios.
- Test long-running graph progression across approval, execution, review, QA, reclaim, retry, repair insertion, and dependent unlocking.
- Keep UI and console tests focused on artifact inspectability, editability, diff visibility, wave preview clarity, and reconciliation transparency instead of brittle presentational details.

## Out of Scope

- Full autonomy without explicit approval boundaries
- Replacing GitHub as the default Operational Tracker
- Forcing users to author large DAG schemas manually before planning can proceed
- Treating a vague or contradictory PRD as automatically runnable just because the user asked for issue generation
- Infinite automatic repair behavior or unbounded recursive bug-loop growth
- Hiding the DAG, loop specs, or reconciliation artifacts behind opaque runtime-only abstractions
- Assuming a fixed provider-specific or thread-specific subagent limit as a durable planning invariant

## Further Notes

- This PRD defines a more concrete operational answer to “almost autonomous vibe coding” for a technical Solo Builder: the user may start from an idea, but the system must convert that idea into a vetted execution graph before broad AFK work begins.
- The core design move is to separate artifact authority by phase:
  PRD for product intent upstream,
  vetted DAG for execution authority downstream,
  loop specs for concrete per-node execution shape,
  GitHub issues for live operational tracking.
- The system should aim for low-friction defaults: generate everything it can, require user approval at the right boundaries, and let the user override any planning artifact directly when the automation gets it wrong.
- The strongest tracer bullet for this PRD is:
  approved PRD -> validated topologically sorted DAG -> GitHub issue synthesis -> approved wave bundle -> compiled Ralph Loop Specs -> parallel-safe execution -> automatic repair insertion -> deterministic graph progression
